### ⚠ G3 内嵌的待决策项：SSE 认证方式（审计中发现）

表 4.10 将问答接口设计为 `GET /api/chat/stream?question=...`，而 4.5.1 要求受保护接口走 `Authorization: Bearer` 请求头。问题在于：**浏览器原生 `EventSource` API 不支持自定义请求头**，两条规范放在一起是无法直接实现的。三个可选方案：

1. **fetch + ReadableStream 手动解析 SSE**（或 `@microsoft/fetch-event-source` 库）——可带 Header，推荐，改动只在前端；
2. Token 放查询参数——实现最快，但 Token 会进访问日志，与安全设计冲突，不推荐；
3. 一次性 ticket 换流——最安全但引入新端点，一周工期下过重。

**建议在 G3 冻结契约时直接采纳方案 1**，并把对应验收写法落在 AC-05。这是「定义不清造成返工」的典型样本——在第四天发现要付出的代价远大于现在。
