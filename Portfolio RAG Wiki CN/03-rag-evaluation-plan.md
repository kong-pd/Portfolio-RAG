## 1. 评测维度与通过线

| 维度 | 指标 | 计算口径 | 通过线（v1 基线目标） |
|---|---|---|---|
| 检索质量 | **hit@5** | 黄金证据（gold_evidence 关键词）所在块出现在 top-5 检索结果中记 1，否则 0 | ≥ 80% |
| 回答忠实度 | **faithfulness**（0–2 人工评分） | 2 = 全部要点有据；1 = 大体有据但含未支撑细节；0 = 关键事实编造 | 均分 ≥ 1.6，且 0 分条目 ≤ 1 |
| 引用准确性 | **citation acc.** | 前端展示的来源确实支撑回答主体内容 | ≥ 90% |
| 拒答正确性 | **refusal acc.** | 无答案型用例必须触发拒答话术且零编造（对应 AC-04.2） | 100% |
| 性能 | 首 token 延迟 | 评测过程顺带记录，取 P50（对应 AC-04.4） | ≤ 3 s |

> 阈值是起点不是终点：第一轮（run_1）以实测定基线，第二轮（run_2）对照通过线决定是否进入打磨。若 hit@5 低，先调检索（阈值、k、分块）；若 hit 高但 faithfulness 低，问题在 Prompt 模板——两类病因不同，不要混着修。

## 2. 黄金集构成（共 28 条）

| 类型 | type 值 | 条数 | 目的 | 示例 |
|---|---|---|---|---|
| 事实型（单块可答） | `factual` | 12 | 验证基础检索 + 生成链路 | 「Access Token 的有效期是多久？」 |
| 综合型（跨块 / 跨文档） | `synthesis` | 6 | 验证多块注入与归纳能力 | 「本系统与 ChatPDF 的核心区别？」 |
| 无答案型 | `no_answer` | 4 | 验证 0.75 阈值拒答，反幻觉 | 「系统支持语音输入吗？」 |
| 多轮型 | `multi_turn` | 3 | 验证 6 轮上下文注入与指代消解（AC-06.4） | 先问分块策略，追问「重叠是多少？」 |
| 边界型 | `edge` | 3 | 口语化、错别字、简繁混杂等鲁棒性 | 「chunking戰略具体是怎麼做的 512？」 |

## 3. 字段说明（对应 CSV 列）

| 列 | 含义 | 填写要点 |
|---|---|---|
| `id` | 编号 Q01… | 按类型分段编号便于筛选 |
| `type` | 上表五类 | factual / synthesis / no_answer / multi_turn / edge |
| `question` | 用户问法 | **用自然口语，不要照抄文档原句**——照抄会让检索退化为关键词匹配，测不出语义能力 |
| `source_doc` | 期望命中的文档 | 无答案型留空 |
| `gold_evidence` | 判 hit@5 的依据 | 写「应命中块包含的关键词 / 短语」，如：512 token、64 重叠 |
| `expected_answer_points` | 标准答案要点 | 列要点不写全文，供 faithfulness 评分对照 |
| `expected_behavior` | answer / refuse | 无答案型 = refuse |
| `pass_criteria` | 该条通过定义 | 默认「要点全覆盖且无编造」，特殊条目单独写 |
| `notes` | 备注 | 来源章节、设计意图 |
| `run_1` / `run_2` | 两轮结果 | 记 `hit=1;faith=2;cite=1` 格式，无答案型记 `refuse=1/0` |

## 4. 种子语料建议

把毕业设计报告四章导出为 **PDF 或 Markdown** 作为首批测试语料——系统范围本就不支持 docx，顺手把「上传 docx 应返回 415」留作 AC-02.3 的天然弹药。报告内容你我都熟，判分快、争议少，适合做回归基线。等真实使用文档（简历、JD、课程笔记等）确定后再补一批贴近实际的条目。建议两套语料长期并存：**报告语料管回归，真实语料管体验**。

## 5. 执行节奏

| 轮次 | 时点 | 范围 | 动作 |
|---|---|---|---|
| 冒烟 | D2 晚（垂直切片打通后） | 10 条子集（factual×6 + no_answer×2 + synthesis×2） | 人工跑，只看 hit@5 与有无幻觉，问题当晚归因 |
| 全量基线 | D4（FR-01~04 AC 达标后） | 28 条 | 结果记入 `run_1`，确立基线 |
| 回归 | 任何影响分块 / 检索 / Prompt 的改动后 | 冒烟子集 | 防止「修一处坏一片」 |
| 二轮全量 | D6 | 28 条 | 记入 `run_2`，对照 §1 通过线，决定进入打磨或继续修 |
| 自动化（stretch） | D7 或后续 | hit@5 可全自动 | pandas 读 CSV → 调 API → 自动判 hit；faithfulness 保持人工。此脚本即「评测看板」后端雏形 |

## 6. 评分纪律

faithfulness 人工评分容易越评越松，三条纪律：对照 `expected_answer_points` 逐点核对，不凭整体印象；回答里出现要点之外的具体数字 / 名词，必须回查检索块，查无实据即扣到 1 或 0；拿不准的条目标记后隔天复评，不当场翻案。
