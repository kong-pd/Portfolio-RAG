# 01 · 系统范围（Scope）

> **用途**：防范围蔓延的主武器不是 in 列表，是 out 列表。
> **状态**：v1.0 定稿（2026-07-10）

## 0. 范围三规则

1. **Out 与 in 同权**：写进 out 的条目连讨论都不必发起，除非走第 3 条。
2. **新想法一律进 `BACKLOG.md`**，附一句"为什么想要"；每个 Gate 结束时统一 triage 一次，平时不看。
3. **范围变更 = 一份新 ADR**：写明加什么、砍什么来换、对 Gate 日期的影响。三样缺一不改。

## 1. In-scope

### P0 —— Gate 1/2 必须，缺一不可投递

**账户与安全**
- 邮箱 + 密码注册 / 登录；BCrypt 存储；JWT 短效 access + refresh
- 全部资源按 user_id 隔离，**含向量检索层的 metadata 过滤**
- 跨用户访问自动化测试（返回 404，不泄露资源存在性）

**文档**
- 上传 PDF / TXT / Markdown；单文件 ≤ 20 MB；单用户 ≤ 50 份（保护免费额度）
- Tika 抽取文本；异步向量化；状态机 UPLOADED → PROCESSING → READY / FAILED
- 文档列表 / 详情 / 删除；删除须同步清理向量与原文件

**问答**
- 自然语言提问；top-k 向量检索；回答附来源引用，来源可展开查看原文 chunk
- SSE 流式输出，事件协议 token / sources / done / error
- 多轮会话（滑动窗口，近 N 条消息，默认 6）；会话列表 / 新建 / 删除
- 知识库外问题明确回答"不知道"（prompt 约束 + 用例验证）
- 中文与英文**文档内容**均可被正确切块、检索、问答

**工程底座**
- `docker compose up` 一键起全套（后端 + Postgres/pgvector）
- OpenAPI（springdoc）+ Postman collection 随仓库提交
- 429 限流的批处理与指数退避重试（免费档生存必需）
- README：Mermaid C4 图、快速启动、30 秒 GIF、ADR 索引

**前端（Gate 2）**
- 三个页面：登录注册、文档（上传 + 状态 + 列表）、聊天（流式渲染 + 来源展开）
- 标准：能用、清爽，不追求精美；桌面浏览器优先

### P1 —— Gate 3

- AWS 单机部署（EC2 或 Lightsail + compose）；文件存储由本地卷切 S3
- GitHub Actions：test → build → 推镜像 → 部署
- Actuator 健康检查；结构化日志
- 基础 API 限流（每用户每分钟提问数上限）
- 会话重命名；失败文档一键重试

### P2 —— Stretch（按性价比排序，做多少算多少）

1. 评测集：≥ 20 条 golden Q&A，命中率 / 忠实度结果表进 README
2. Hybrid search：pgvector + Postgres 全文检索，RRF 融合
3. 安全加固：上传 magic bytes 校验、文档内容 prompt injection 缓解
4. Postgres RLS 版隔离（与应用层过滤对比，写成 ADR）
5. 轻量 rerank（可选，评测集先行才有意义）

## 2. Out-of-scope（逐条附理由）

### 产品功能类

| 不做 | 理由 |
|---|---|
| 团队 / 组织 / 共享知识库 / 角色权限（RBAC） | 多租户协作是另一个项目的体量；单用户强隔离已足够支撑面试叙事 |
| 文档公开分享链接 | 引入匿名访问的全新安全面，收益却只是演示花活 |
| OCR、扫描件 PDF、图片内文字 | OCR 是独立技术栈，投入产出比极低；demo 一律用文本型 PDF |
| 表格 / 图表结构化抽取与问答 | RAG 处理表格是前沿难题，12 周内做不出可信版本 |
| DOCX / PPTX / EPUB / 网页抓取等更多格式 | Tika 技术上能做，但"再加一种格式"是最经典的蔓延起点；三种格式足够讲完整故事 |
| Agent / 工具调用 / 联网检索 / 多知识库路由 | 把 RAG 做深胜过把 agent 做浅；面试深挖时半吊子 agent 是负资产 |
| 文档在线编辑、批注、引用高亮定位回 PDF 原页码 | 展示来源 chunk 文本已达差异化目的；PDF 坐标级定位成本极高 |
| 通知（邮件 / 推送）、管理后台、数据导出 | 与核心叙事无关的纯工作量 |
| 计费 / 订阅 / 配额购买 | 作品集不需要商业化模块 |
| 语音输入 / 输出 | 与核心叙事无关 |

### 账户类

| 不做 | 理由 |
|---|---|
| OAuth 第三方登录（Google 等） | 面试要考的是你手写的 JWT 全流程，OAuth 反而把它遮住 |
| 邮箱验证、找回密码 | 真产品必须、作品集示范无益；README 一行注明"演示项目简化" |
| 用户资料页、头像 | 纯工作量 |

### 工程类

| 不做 | 理由 |
|---|---|
| 微服务拆分、Kubernetes、服务网格 | ADR-002 已裁决单体；"什么时候不该拆"本身就是面试答案 |
| 高可用、多副本、自动扩缩容 | NFR 明确单机目标；伪装的"生产级"经不起追问 |
| 多区域部署、CDN | 同上 |
| 模型微调、自训 embedding | 完全不同的技能树 |
| 多语言 UI / i18n | 界面固定英文（面向面试官）。注意：这与 P0 的"支持中文文档内容"是两回事 |
| 移动端 App | 移动网页只保证不崩，不做适配 |

## 3. 灰色地带裁决记录

| 条目 | 裁决 | 理由 |
|---|---|---|
| 会话删除 vs 重命名 | 删除 P0，重命名 P1 | 删除涉及数据清理完整性（消息、上下文），必须顺手做对；重命名纯 UX |
| 引用展开显示原文 chunk | P0，不可砍 | 这是与红海 RAG 项目的分界线之一 |
| 上传进度条 / 拖拽上传 | 组件库自带就用，不专门开发 | 纯 UX，不值专门投入 |
| 失败文档重试按钮 | P1 | Gate 1 阶段删掉重新上传即可 |

## 4. 与其他文档的关系

SCOPE 管"做不做"；每条 P0 在 `02-REQUIREMENTS.md` 有对应 FR 编号与验收标准，管"做到什么算做完"；执行粒度在 `04-TASKPLAN.md` 的任务卡里。三层对不上时，以 SCOPE 为准并触发第 0 节第 3 条。
