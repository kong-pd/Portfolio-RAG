# ADR-002 · 架构形态：模块化单体，不拆微服务

**状态**：Accepted（2026-07-10）

**背景**：单人开发、约 12 周预算；作者已有微服务项目（ColdWatch），本项目的差异化在"判断力"而非重复演示拆分。

**选项**：微服务；无边界单体；模块化单体（package by feature + 边界约束）。

**决定**：模块化单体。四模块 `auth / document / ingestion / chat`，模块间只经 service 接口调用，禁止横向 import（P1 起以 ArchUnit 测试强制）。理由：无独立扩缩容需求；文档删除需要跨"关系数据 + 向量"的强事务；单人维护下拆分只有成本没有收益。模块边界即日后拆分的预留缝。

**后果**：单点故障，与 NFR-3 单机目标一致。面试话术现成："什么时候会拆——团队超过一人、或 ingestion 吞吐需要独立伸缩时，第一个拆出去的是 ingestion。"
**重审触发**：ingestion 长任务持续挤占问答线程池且线程池隔离调参无效 → 先引入队列解耦，仍不拆服务。
