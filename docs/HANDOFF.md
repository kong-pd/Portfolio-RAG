# HANDOFF · 模型交接手册

> 给人看的操作手册：如何在新窗口、新模型处继续本项目。
> 核心原则一句话：**交接的载体是仓库，不是聊天记录。** 文档包、任务卡勾选、git 历史加起来就是项目的全部状态；prompt 只是指向状态的指针。指针可以很短，状态必须很全——这就是我们把判断全部前置进文档的原因。

## 可靠性排序

1. **首选：Claude Code 直接在仓库里开工。** 它自动读根目录 CLAUDE.md，能自己读 docs/、跑 `mvn test`、查 `git log`——状态零拷贝、零转述、零漂移，交接成本趋近于零。
2. **次选：网页对话 + context kit。** 手工粘贴 00-BRIEF + 当前任务卡 + 卡内引用的 ADR/03 章节 + 状态三行。缺点：模型无法亲自验证仓库现状，验收命令要你替它跑、结果贴回去。
3. **不要做的三件事：** 把五份文档全文塞进每次会话（上下文噪音淹没重点）；同一会话连做多张卡（上下文腐化，质量随长度下降）；口头转述报错（永远贴原文）。

## Prompt A · Claude Code（新模型第一次上岗）

```
读根目录 CLAUDE.md，执行其中"接手审计"五步并输出审计报告。
审计报告我确认后，领取 docs/04-TASKPLAN.md 中第一张 [ ] 状态的卡，
按 04 §0 规则开工：先给实现计划，等我确认再动手。
```

之后的日常会话只需要一行：

```
读 CLAUDE.md。本次做 T<X.Y>，按规则先给计划。
```

## Prompt B · 网页对话（模型无法访问仓库时）

粘贴顺序：下面模板 → 00-BRIEF 全文 → 当前任务卡 → 卡内引用的 ADR / 03 章节。

```
【角色】你是 Portfolio RAG 项目的实现工程师，接手一个进行中的项目。
【状态】当前卡：T<X.Y>；最后 commit：<粘贴 git log --oneline -1 输出>；
已知卡点：<没有就写"无">。
【规则】
1. 只做本卡范围，需越界先停下报告；
2. 先给实现计划（文件清单+要点），等我确认再写代码；
3. 完成后逐条对照卡内验收清单自检，告诉我每条怎么验证；
4. 讲解关键代码，并用卡内理解题考我，我答不出就讲到我会；
5. 禁止修改验收标准、禁止大规模重构，范围外想法只记 BACKLOG。
【随附材料】00-BRIEF、当前任务卡、相关 ADR 章节如下：
```

## 收工三行（每次结束前更新到 docs/LEARNING.md 顶部）

```
当前卡与状态：
最后 commit：
卡点 / 待办：
```

这三行是 Prompt B 的【状态】栏的直接来源。写它花三十秒，省下一次新会话开头十分钟的"我做到哪了"考古。

## 升级点射（什么时候值得调用高阶模型）

架构级变更提案、诡异的集成 bug（两小时规则触发后）、auth 代码安全 review、Phase 3 的 RAG 质量调优、Gate 2 前的项目模拟面试。点射时同样带 context kit——高阶模型一样不会读心。
