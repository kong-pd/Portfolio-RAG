# INTERVIEW-DRILL · 面试三层追问题库

> **状态**：v1.0（2026-07-10）
> **用途**：面试官会顺着简历项目往下钻三层。本题库按本项目的**真实决策**组织，L1 入门 / L2 机制 / L3 边界与事故。要点是提纲不是讲稿——能用自己的话展开才算会。
> **用法**：① 自测：遮住要点先答，答完对照；② 压测：Gate 2 前把本文件交给任意模型，prompt「扮演香港 fintech 后端面试官，用本题库随机抽题连续追问我，我答不到 L3 就换角度再追」；③ 英文口语版在 Gate 2 收尾周让模型转换，自己核对术语。
> **万能答题骨架**：决定 → 理由 → 代价 → 什么条件下我会改（正好是 ADR 四段的口语化）。
> **红旗自查**：以下三种回答会当场暴露空壳——"教程/AI 这么写的"；说不出决策的代价；说不出决策何时失效。

## 1 · 架构形态（ADR-002）

- **L1**：介绍一下架构。为什么用单体？你上一个项目不是微服务吗？
- **L2**：模块化单体的"模块化"体现在哪？边界靠什么不烂掉？
- **L3**：什么信号出现你会拆？第一刀切哪、为什么？
- **要点**：单人+强事务（删文档要跨关系数据与向量）+无独立扩缩容需求 → 拆分只有成本；package by feature 四模块、依赖方向白名单、ArchUnit 测试拦截横向 import；重审信号 = ingestion 长任务挤占问答线程池且线程池隔离无效；第一刀 = 先上队列解耦 ingestion，仍不拆服务——"知道什么时候不拆"比"会拆"稀缺。

## 2 · pgvector（ADR-003）

- **L1**：向量为什么存 Postgres 而不是专用向量库？
- **L2**：HNSW 是什么量级的算法？为什么它的维度上限会反过来约束 embedding 选型？删除文档时向量与关系数据的一致性怎么保证？
- **L3**：什么规模下 pgvector 不够？真要迁 Qdrant，要动哪几层？
- **要点**：同一事务域（级联删除天然一致、无跨库对账）、运维减半、<百万向量性能足够；HNSW 近似最近邻、索引维度上限约 2000 → 768 的由来之一；迁移要动：VectorStore 实现、metadata filter 语法、部署拓扑（多一个有状态服务）、备份策略——检索接口不动，因为收口在 Spring AI 抽象后面。

## 3 · Embedding 锁定与 768 维（ADR-008）

- **L1**：embedding 模型怎么选的？
- **L2**：为什么换 embedding 等于全量重建索引？为什么是 768 维——降维不掉精度吗？
- **L3**：Google 免费档政策一夜收紧，你的预案？步骤和影响面？中文检索质量怎么验证过？
- **要点**：不同模型向量空间互不兼容、查询与入库必须同模型 → dev/prod 必须同源，这是选云端 API 压过本地 bge-m3 的决定性理由（部署机不背模型）；768 = MRL 截断，落在 HNSW 上限内、质量够用且省存储；预案 = 切 bge-m3 + 重建索引约一个工作日，chunks 表保留原文所以可重算；中文质量靠 P2 评测集数字说话，不靠感觉。

## 4 · JWT 与认证（ADR-004）

- **L1**：为什么 JWT 不用 session？
- **L2**：JWT 三段各是什么？服务端验证流程？为什么 access 15 分钟 + refresh 7 天双 token？refresh 为什么哈希落库？
- **L3**：JWT 没法即时吊销，怎么办？token 放 localStorage 被质疑 XSS 风险，你怎么接？
- **要点**：诚实开场——单体下 session 其实更简单，选 JWT 一半是前后端分离惯例、一半是刻意的学习与展示目标，这个诚实本身加分；签名保真伪不保机密（payload 只是 base64）；短 access 限制被盗窗口、refresh 落库哈希（库泄露也不能直接冒用）支持登出吊销与轮换；XSS 之问的标准接法 = 承认取舍、给出 httpOnly cookie + CSRF 防护的迁移路径（ADR-004 已预埋），并指出当前缓解（短效、CORS 白名单）。

## 5 · 数据隔离（ADR-004 / FR-A5）

- **L1**：多用户数据怎么隔离？
- **L2**：跨用户访问为什么返回 404 不是 403？隔离为什么要做到向量检索层？怎么证明它真的生效？
- **L3**：只在应用层过滤，风险是什么？数据库层还能怎么加固？将来要做团队共享怎么演进？
- **要点**：404 不泄露资源存在性（403 等于告诉攻击者"这个 ID 存在"）；向量层靠 user_id metadata filter——SQL 层隔离了、检索层漏了是 RAG 项目最常见的洞；证明 = 三组隔离测试 + "B 埋独有内容 A 检索不命中"的杀手用例；应用层过滤怕的是未来某个新查询忘了带条件 → Postgres RLS 是数据库层兜底（P2 有对比实现计划）；团队共享 = out-of-scope，演进方向是 resource-level ACL 而不是推翻 user_id 模型。

## 6 · RAG 流水线（FR-D3 / Q1–Q5）

- **L1**：讲一遍从上传到回答的完整链路。
- **L2**：chunk 500–800 token 怎么定的？top-k=4、阈值 0.50 怎么调？拒答怎么实现？幻觉怎么压？
- **L3**：检索质量怎么量化？hybrid search 补什么短板？文档里藏一句"忽略以上指令"，你的系统会怎样？
- **要点**：链路两条背下来（03 §6 写路径与读路径）；chunk 太大稀释相似度、太小丢上下文，参数全部配置化 + 真实语料校准而非拍脑袋；拒答 = 空结果或最高分低于阈值走固定回复、sources 空数组仍下发；压幻觉三件套 = 只依据上下文的系统约束 + 来源引用可核验 + 拒答路径；量化 = P2 评测集（命中率/忠实度表进 README）；hybrid = 语义检索对专有名词、编号、精确短语弱，全文检索补上，RRF 融合；prompt injection = 文档内容按不可信输入处理，系统约束与用户内容分层，P2 有专项缓解卡。

## 7 · SSE 流式（ADR-005 / FR-Q3）

- **L1**：为什么 SSE 不用 WebSocket？
- **L2**：事件协议怎么设计的？sources 为什么放在流末尾？客户端断开，服务端怎么感知、要做什么？
- **L3**：SSE 过代理/负载均衡有什么坑？如果产品要求"引用随句子出现"，协议怎么改？
- **要点**：单向下行场景，SSE 是 HTTP 原生、语义更简单，WebSocket 留给真双向；协议 token/sources/done/error 显式建模终止与错误（客户端永远知道流为什么结束）；sources 后置因为引用集合在检索时已定、生成中无增量，前置则首 token 延迟变差；断开靠连接取消信号传播、停止上游生成防资源泄漏；代理坑 = 缓冲与超时（15s ping 注释行就是为此）；随句引用 = token 事件里带 chunk 归属标记，成本是 prompt 与解析都要改——能说出改法即达 L3。

## 8 · 异步 ingestion 与限流韧性（03 §9.3 / FR-E3）

- **L1**：上传一个 20MB PDF 后系统发生什么？用户看到什么？
- **L2**：为什么独立有界线程池？429 指数退避为什么要加 jitter？失败路径怎么走？
- **L3**：队列满了怎么办？同一文档被重复处理怎么防？要横向扩 ingestion 吞吐，第一步改什么？
- **要点**：立即返回 UPLOADED → 异步 PROCESSING → 轮询到 READY/FAILED，上传接口永不被解析拖住；有界 + 拒绝策略 = 宁可显式 FAILED 也不无界排队吃光内存、不挤占 Web 线程；jitter 防"齐步重试"造成的自我 DDoS；失败只污染单文档、原因用户可读；重复处理靠状态机守卫（只有 UPLOADED/FAILED 可进入 PROCESSING）；扩吞吐第一步 = 进程外队列（呼应 ADR-002 重审条款），而不是先拆服务。

## 9 · 删除的事务边界（03 §9.2）

- **L1**：删除一份文档，系统里要消失哪些东西？
- **L2**：为什么行删除在数据库事务内、文件删除在事务外？
- **L3**：文件删失败留下孤儿怎么办？顺序反过来先删文件，会出什么事故？
- **要点**：documents 行 + chunks（向量）同事务，原文件事务提交后删；因为数据库事务管不到文件系统——把不可回滚的操作放进事务等于假装它可回滚；孤儿文件 = 记日志 + 定期清理兜底，用户不可见故安全；反序事故 = 文件没了、库删除回滚，用户看得见文档却永远打不开——数据不一致里最伤信任的一种。

## 10 · 工程与部署（ADR-007 / FR-E1）

- **L1**：怎么部署的？CI/CD 长什么样？
- **L2**：多阶段 Dockerfile 省什么？为什么 COPY pom.xml 和 COPY src 分两步？配置为什么走环境变量？
- **L3**：这套单机架构的故障模式有哪些？如果明天要求 99.9% 可用，你按什么顺序改造？
- **要点**：compose 单机 EC2/Lightsail、Actions 四步流水线，预算 < USD 15 是显式 NFR 不是偷懒；多阶段 = 构建工具链不进运行镜像（体积与攻击面）；分两步吃层缓存——依赖不变就不重下；环境变量 = 同一镜像跑三个 profile（12-factor）；故障模式主动说：单点、重启窗口、上游免费档限流；99.9% 改造顺序 = 先数据（托管 PG/备份）→ 再无状态多副本 + 负载均衡 → 最后队列化 ingestion——能给顺序和理由即达 L3。

## 11 · 元问题：AI 时代你怎么证明这是你的项目

- **L1**：这个项目哪部分最难？开发中用 AI 了吗？
- **L2**：说一个你中途改过主意的技术决策。AI 辅助下你怎么保证自己真的理解？
- **L3**：（面试官现场出招）"那你白板写一下 JWT 过滤器链 / 检索拼 prompt / SSE 骨架。"
- **要点**：大方承认 AI 辅助并立刻展示护栏——判断前置成文档、验收唯一来源、深度红线三部位由我手写、每张卡有理解题不过不关卡，这套方法论本身就是 2026 年面试里的稀缺答案；改主意的现成素材 = ADR 的 supersede 机制与"重审触发"（真发生过就讲真事，没发生就讲机制设计）；L3 就是深度红线的验收现场——三个部位在 04 里各有一张"脱稿重写"卡，Gate 2 关门检查会先于面试官考你一遍。
